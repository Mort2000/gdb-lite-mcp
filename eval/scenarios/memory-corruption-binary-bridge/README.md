# memory-corruption-binary-bridge

## Purpose

Evaluates memory-corruption triage across visible C integration code and a
black-box binary API. The efficient route is to inspect the vendor API boundary,
the producer's reported required length, and the first write that corrupts the
adjacent canary.

## User-Visible Project

The workspace contains `src/import_job.c`, `src/vendor_manifest.h`,
`data/manifest-feed.bin`, and `bin/import-job`.

## Hidden Build Inputs

`private/vendor_manifest.c` is compiled as a linked vendor object with
optimization and without debug information, then stripped. The binary input
`data/manifest-feed.bin` is generated by `private/make_manifest_feed.py`.

## Expected Diagnosis

`vendor_manifest_key_from_file` parses `manifest-feed.bin` and reports a
required key length larger than the fixed 48-byte `ImportJob.key` buffer. It
fills all 48 bytes of `key` and leaves no `NULL` terminator. The visible
caller ignores the required length and passes the truncated buffer to
`vendor_manifest_stamp`, which treats the buffer as an in-place C string and
writes past `key[47]`; the first out-of-bounds write lands at `key[48]`, which
is `canary[0]`.

## Anti-Cheat Notes

The prompt identifies the black-box vendor boundary and binary feed, but does
not disclose the required length mismatch, the missing `NULL` terminator, or
the adjacent canary overwrite. The private vendor source and feed generator are
not installed into the model workspace.
